# Phase 15 Final Plan

Provenance: generated by the `/hyperplan` adversarial team and sequenced by the plan agent (`ses_0a5eb7284ffep5SwmNw2ZbTNUh`).

## Context

Phase 15 adds prompt duplication, derived prompt drafts, prompt template CRUD/apply workflows, and lineage display while preserving the Phase 11-14 guardrails.

The final plan intentionally replaces the draft's alternatives with fixed MVP decisions. The goal is an implementation plan that a worker can execute without re-designing the data model, IPC surface, or renderer state semantics.

## Fixed Decisions

- Do not add `prompt_asset_lineage` for MVP.
- Extend `prompt_assets` with nullable `parent_prompt_version_id` and nullable `derivation_type`.
- Keep existing `parent_prompt_id`.
- MVP derivation enum is exactly `duplicate | derived`.
- Do not persist `templated_from` or `improved_from_review` relationships in Phase 15.
- When a source asset or source version is deleted, `parent_prompt_id` and `parent_prompt_version_id` become `NULL`; `derivation_type` may remain.
- General prompt asset create/update APIs must not mutate lineage fields.
- Only dedicated duplicate/derived commands set `parent_prompt_id`, `parent_prompt_version_id`, and `derivation_type`.
- Duplicate, derived save, and lineage navigation are same-project only. Cross-project duplication, derivation, and navigation are out of scope.
- Add one main-process atomic command, `createPromptWithInitialVersion`, used by normal save, duplicate, and derived save.
- `createPromptWithInitialVersion` creates asset, version 1, current version pointer, optional tags, and FTS/index entry in one transaction.
- FTS/index updates are transactional for Phase 15. A create operation must not report success unless asset, version, current pointer, tag links, and FTS row all commit.
- Duplicate title policy is `Copy of {sourceTitle}`.
- Duplicate creates version 1 from the selected source version or source current version.
- Duplicate copies prompt content fields but never copies `quality_score` or `prompt_quality_reviews`.
- Duplicate copies tags by default; caller may pass `copyTags: false`.
- Derived action from a `PromptVersion` seeds `PromptCompilerPanel` draft state only.
- Derived action creates no DB rows until explicit save.
- Clearing derived provenance removes only metadata, not content.
- Saving a derived draft creates a new asset and version 1 with `derivation_type = derived`.
- Prompt templates use a new `prompt_templates` table only.
- `prompt_templates` columns are exactly `id`, `name`, `description`, `source_prompt_asset_id`, `source_prompt_version_id`, `scenario`, `target_agent`, `template_body`, `created_at`, `updated_at`.
- Do not store template variables or template tags in DB.
- Template source IDs are immutable after create and use `ON DELETE SET NULL`.
- Template variables are computed from `template_body`.
- Prompt template preview/extract are local pure utilities, not IPC.
- Prompt template IPC includes only create, list, get, update, duplicate, delete, and createFromVersion.
- Selecting a prompt template or editing variable preview state does not mutate effective draft, persist data, call LLM, or mark stale.
- Only explicit Apply with confirmation mutates the output draft.
- Apply increments a draft/output revision token, clears stale analysis/compile answers/suggested tags, and makes Phase 14 review stale through existing snapshot comparison.
- Applied output is immutable even if the template is later edited or deleted.
- Saving template-applied output must use a real saveable version payload and must not invent `qualityScore`.
- Saving remains disabled until title and non-empty `originalInput` are present where current contracts require them.
- Template apply never auto-overwrites `originalInput`.
- `getLineage` returns nullable parent summary and child summaries in one API. Do not add separate `listChildren`.
- Lineage UI uses cards/lists only. No graph/canvas library.

## Not In Scope

- Prompt execution.
- `prompt_runs`, `agent_runs`, `execution_results`, `validation_results`, or `run_logs`.
- Codex, Claude Code, Cursor, generic agent, CLI, or external app execution.
- Codex OAuth.
- External repo scanning.
- `repo_path` file reads or filesystem scans.
- `globalShortcut`.
- `window.prompter.appEvents.*`.
- `window.prompter.shortcuts.*`.

- Prompt template marketplace, remote template download, cloud sync, graph visualization, or collaboration.
- Persisted `templated_from` or `improved_from_review` lineage.

## Commit Strategy

Use atomic commits in this order:

1. `Add phase 15 guardrail and contract tests`
2. `Add prompt derivation schema and templates migration`
3. `Add atomic prompt creation and derivation services`
4. `Wire prompt template persistence and IPC`
5. `Add local prompt template utilities`
6. `Add derivation and template apply state`
7. `Add prompt template and lineage UI`
8. `Add phase 15 integration coverage`
9. `Finalize phase 15 QA fixes`

Commit rules:

- Do not mix schema/migration changes with renderer UI changes.
- Do not commit generated local DB files, `.codegraph/`, `.omo/`, logs, build output, or runtime state.
- Before committing, inspect `git status`, `git diff`, and `git log --oneline -10`.
- Stage only intended files.
- Never amend unless explicitly requested.

## Success Criteria

- Prompt assets can be duplicated from a selected/current version in the same project.
- Duplicates start at version 1 and record `duplicate` lineage using existing asset columns.
- Derived drafts can be created from a version without DB writes.
- Derived drafts save explicitly as new same-project assets with `derived` lineage.
- Prompt templates can be created, listed, searched, filtered, viewed, updated, duplicated, deleted, and created from a prompt version.
- Template variables are extracted/rendered locally and safely.
- Template selection/preview does not mutate draft, persist, call LLM, or mark stale.
- Template apply is explicit, confirmable, stale-safe, revision-safe, and saveable without fake quality scores.
- Lineage UI shows parent, children, and deleted-source states.
- Search/FTS updates are transactionally consistent with creates.
- Restart preserves templates and lineage.
- Phase 11-14 guardrails remain intact.
- All verification commands pass.
