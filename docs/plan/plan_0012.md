# Phase 12 Implementation Plan: Harness Template Management UI

## Goal

Implement Phase 12 harness template management and prompt compiler integration for Prompter.

Phase 12 adds:

- Default harness template seeding.
- Harness template CRUD plus duplicate.
- List, filter, search, select, edit, duplicate, and delete UI.
- PromptCompilerPanel harness selector.
- Static compiler integration through safe placeholder substitution.
- LLM compiler integration where the selected harness is untrusted additional guidance only.
- Stale compiler state reset when `harnessTemplateId` changes.

Before this plan was written, `docs/plan/plan_0012.md` existed as an empty file.

## Hard Guardrails

Preserve Phase 11 quick capture behavior:

- No `globalShortcut`.
- No `window.prompter.appEvents`.
- No `window.prompter.shortcuts`.
- No `quick_capture_*` settings.
- No `QuickCaptureSettingsSchema`.
- No `RegisterGlobalShortcutInputSchema`.
- No clipboard history.
- No background clipboard watch.
- No trim, cleanup, blank-line cleanup, or control-character cleanup of clipboard text.
- No trim, cleanup, or mutation of `originalInput` as a side effect of harness selection.
- No auto LLM analyze.
- No auto LLM compile.
- No auto-save.
- No persistence before explicit user save.

Preserve renderer isolation:

- Renderer must not import DB, Drizzle, SQLite, filesystem, path, process, `ipcRenderer`, or Electron internals.
- Renderer must use only `window.prompter` from the typed preload bridge.
- All IPC inputs must be Zod-validated in `electron/ipc-contract.ts`.
- IPC must route through bridge, preload, handlers, services, and repositories.

Preserve compiler safety:

- User `templateBody` is untrusted text.
- Do not execute template content.
- Do not use `eval`, `new Function`, dynamic import, shell execution, or external process execution.
- User harness text must not replace the LLM system prompt.
- User harness text must not override required JSON schemas.
- User harness text must not bypass Zod output validation.
- Static rendering must be simple placeholder substitution only.

Do not add:

- Prompt execution.
- Codex CLI launching.
- Claude Code launching.
- Cursor launching.
- External app control.
- `prompt_runs`.
- `agent_runs`.
- `execution_results`.
- `validation_results`.
- `run_logs`.
- Cloud sync.
- Marketplace.
- Remote templates.
- Settings expansion except where already required by existing patterns.

## Static Template Renderer

Add a pure renderer utility:

- `renderer/src/lib/prompt-compiler/harness-template-renderer.ts`

Supported placeholders:

- `{{title}}`
- `{{originalInput}}`
- `{{scenario}}`
- `{{targetAgent}}`
- `{{projectContext}}`
- `{{techStack}}`
- `{{constraints}}`
- `{{acceptanceCriteria}}`
- `{{validationCommands}}`
- `{{additionalNotes}}`

Rules:

- Replace only allowlisted placeholders.
- Preserve all whitespace in template and values.
- Preserve Markdown, code fences, and diff text.
- Do not execute expressions.
- Leave unknown placeholders unchanged and return warnings.
- Return `{ content, warnings }`.

## Static Compiler Integration

Update:

- `renderer/src/lib/prompt-compiler/types.ts`
- `renderer/src/lib/prompt-compiler/llm-compiler-flow.ts`
- `renderer/src/lib/prompt-compiler/static-prompt-compiler.ts`
- `renderer/src/hooks/use-prompt-compiler-panel.ts`

Add to `PromptCompilerInput`:

- `harnessTemplateId?: string | null`

Static compile behavior:

- If no harness is selected, keep existing output.
- If a harness is selected and loaded, render its `templateBody`.
- Use simple placeholder substitution only.
- Preserve `originalInput` exactly in rendered content.
- Preserve existing compiled result metadata.
- Missing required fields from `requiredFields` should produce warnings, not block compile.
- Unknown placeholders should produce warnings, not execute.
- If selected harness is deleted or unavailable, show recoverable message and fall back to the existing default static compiler.

Fix whitespace-sensitive behavior:

- Do not store or render `draft.originalInput.trim()` as the original input.
- Use trim only for blank checks.

## Atomic Commit Strategy

1. `Lock harness template IPC contract`
   - IPC schemas, bridge methods, list filter, duplicate, compiler `harnessTemplateId` tests.

2. `Add harness seed and repository operations`
   - Default seed, filtering, duplicate, persistence tests.

3. `Render harness templates safely`
   - Placeholder renderer and static compiler integration.

4. `Pass harness guidance to LLM compiler`
   - Main-process harness lookup and untrusted guidance integration.

5. `Add harness template management UI`
   - Sidebar section, manager/editor, selector, stale state reset.

6. `Cover Phase 12 UI and guardrails`
   - Playwright coverage, quick capture regression, forbidden-feature scans.

Keep commits atomic. Do not mix schema/IPC, compiler, and renderer work unless required by TypeScript dependencies.
