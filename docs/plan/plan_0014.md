# Phase 14 Implementation Plan: Prompt Quality Reviewer

## 1. 목표

Phase 14는 저장된 `PromptVersion` 또는 현재 Prompt Compiler draft의 `compiledPrompt`를 대상으로 “프롬프트 지시문 품질”을 평가하는 Prompt Quality Reviewer를 구현한다.

이 기능은 프롬프트를 실행하지 않는다. Codex, Claude Code, Cursor, git, shell command, 외부 앱, 외부 repo, `repo_path` 파일 시스템을 호출하지 않는다. 실행 결과 저장소도 만들지 않는다.

MVP 목표는 다음이다.

- API Key 없이 동작하는 Local heuristic reviewer 구현
- 저장된 `PromptVersion` 품질 리뷰
- 현재 draft `compiledPrompt` 품질 리뷰
- `prompt_quality_reviews` 테이블 기반 리뷰 저장
- 사용자의 명시적 액션으로만 `prompt_versions.quality_score` 반영
- 명시적 클릭으로만 실행되는 main-process LLM review 또는 안전한 placeholder
- Phase 11 quick capture, Phase 12 harness, Phase 13 project context guardrail 유지

## 2. 범위와 비범위

### MVP 범위

- Prompt quality schema/type/grade utility
- deterministic local heuristic reviewer
- `prompt_quality_reviews` migration/repository/service
- typed IPC + preload bridge + renderer-safe types
- saved PromptVersion quality panel
- PromptCompilerPanel draft review entry
- stale review snapshot 처리
- review 저장, 최신 review 조회, 명시적 score 반영
- review mode/provenance/timestamp/stale/fresh/saved/unsaved UI 표시
- forbidden surface guardrail tests/scans
- export 기본 동작 유지: quality review metadata 자동 포함 금지

### Stretch / 선택 범위

- LLM review full service
- review history UI 확장
- `improvedPromptDraft` diff viewer
- 저장된 PromptVersion의 “개선안을 새 버전으로 저장”
- draft의 명시적 “preview 후 suggestion 적용”
- prompt card/search result quality badge
- Markdown export의 “include quality review” opt-in checkbox

### 비범위

- prompt execution
- Codex/Claude Code/Cursor 실행
- git 실행
- external repo scan
- `repo_path` 기반 파일 읽기
- globalShortcut
- `window.prompter.appEvents.*`
- `window.prompter.shortcuts.*`
- `quick_capture_*` settings
- clipboard history/background watch
- `prompt_runs`, `agent_runs`, `execution_results`, `validation_results`, `run_logs`
- automatic LLM review
- automatic save
- automatic `quality_score` update
- automatic prompt overwrite
- score sorting/filtering MVP
- review metadata 자동 export 포함

## 3. 핵심 결정

| 주제 | 결정 |
|---|---|
| 리뷰 의미 | 프롬프트 실행 가능성 자체가 아니라 “지시문 명세 품질”만 평가한다. |
| MVP 엔진 | Local heuristic reviewer를 먼저 구현한다. |

| 기존 score 충돌 | compiler `qualityScore`, DB `quality_score`, review `overallScore`를 구분한다. |
| stale state | draft review는 입력 snapshot에 묶고 변경 시 stale 처리한다. |
| improvedPromptDraft | 저장된 버전 overwrite 금지. 새 PromptVersion 저장만 허용, draft 적용은 explicit preview/diff 후 stretch. |

| export | quality review metadata는 기본 export에 자동 포함하지 않는다. |

## 5. Acceptance Criteria

### Task 1

- valid review result schema test가 존재한다.
- invalid score, invalid grade, invalid severity, invalid reviewMode를 거부한다.
- empty `compiledPrompt`, empty `promptVersionId`를 거부한다.
- forbidden schema/API가 없는지 contract test로 확인한다.

### Task 2

- `PromptQualityReviewMode`, `PromptQualityGrade`, `PromptQualityDimensionScores`, `PromptQualityIssue`, `PromptQualitySuggestion`, `PromptQualityReviewSnapshot`, `PromptQualityReviewResult` schema가 존재한다.
- `ReviewDraftInput`, `ReviewVersionInput`, `SavePromptQualityReviewInput`, `ApplyQualityScoreToVersionInput`, `SaveImprovedPromptAsNewVersionInput`, `ListPromptQualityReviewsInput` schema가 존재한다.
- grade mapping은 90/75/60/40 기준을 따른다.
- `ambiguityRisk`는 inverse/risk semantics로 처리한다.
- renderer-safe type만 공유한다.

### Task 3

- API Key 없이 local review가 동작한다.
- 동일 input에는 deterministic result를 반환한다.
- input string/object를 mutate하지 않는다.
- Markdown/code fence/diff/whitespace를 보존한다.
- Objective, Context, Task, Scope, Constraints, Acceptance Criteria, Validation, Working Instructions, Final Response Format 계열 section을 탐지한다.
- 빈 boilerplate section과 vague acceptance criteria를 penalize한다.
- `overallScore`, `grade`, `dimensionScores`, `strengths`, `issues`, `suggestions`, `missingSections`, `warnings`, `recommendedClarifyingQuestions`, “Why this score?” 설명을 반환한다.

### Task 4

- `prompt_quality_reviews` table이 추가된다.
- JSON fields는 write 시 stringify, read 시 parse + Zod 검증한다.
- corrupted JSON은 recoverable error/safe failure로 처리한다.
- create/list/latest/get/apply score repository 함수가 있다.
- review 저장은 `prompt_versions.quality_score`를 자동 갱신하지 않는다.
- explicit apply action만 `quality_score`를 갱신한다.
- migration/schema test에서 forbidden run/result table이 없음을 확인한다.

### Task 5

- IPC 변경은 `electron/ipc-contract.ts`에서 시작한다.
- `window.prompter.promptQuality.*` namespace만 renderer에 노출한다.
- renderer는 DB/OpenAI/fs/path/process/safeStorage/ipcRenderer에 직접 접근하지 않는다.
- main handler는 payload를 Zod parse 후 service를 호출한다.
- saved version review는 main process에서 saved prompt data를 읽는다.
- prompt body/API key는 log하지 않는다.

### Task 6

- LLM review는 explicit click으로만 실행된다.
- OpenAI SDK는 main process에서만 사용된다.
- API Key missing은 recoverable error이며 local review 가능성을 안내한다.
- LLM prompt는 실행/도구 호출/repo scan/code modification 금지를 명시한다.
- LLM output은 JSON + Zod schema로 검증한다.
- malformed/schema-invalid output은 recoverable error state가 된다.
- 범위가 커지면 service skeleton/UI placeholder까지만 구현하고 local MVP를 우선 완료한다.

### Task 7

- UI는 no compiled prompt, no saved review, local running, LLM running, API key missing, schema failure, stale review, saved-but-score-not-applied, score applied, improved draft available state를 표시한다.
- top-level UI는 grade, mode, source, timestamp, saved/unsaved, fresh/stale, top blockers, missing sections, “Why this score?”, next action을 보여준다.
- detailed section은 dimension scores, strengths, issues, suggestions, warnings, clarifying questions, improved preview를 expandable로 보여준다.
- UI copy는 “prompt instructions quality review” 의미를 분명히 하고 execution/validation/run으로 오해될 label을 피한다.
- LLM button은 prompt text가 OpenAI로 전송될 수 있음을 고지한다.

### Task 8

- saved PromptVersion detail에서 latest review를 볼 수 있다.
- Local review 실행, review 저장, explicit score 반영이 가능하다.
- 저장된 `compiled_prompt`는 자동 수정되지 않는다.
- `improvedPromptDraft`는 preview-only가 기본이다.
- 저장된 PromptVersion에서는 overwrite 대신 “개선안을 새 버전으로 저장”만 허용한다.
- 새 version 저장을 구현하면 기존 Phase 6 version creation flow를 재사용하고 version number는 main process에서 계산한다.

### Task 9

- draft review는 compiled prompt가 있을 때만 가능하다.
- draft review는 자동 저장하지 않는다.
- snapshot은 `compiledPrompt`, `originalInput`, `scenario`, `targetAgent`, `harnessTemplateId`, `projectContextProfileId`, `includeProjectContextProfile`, manual context, constraints, acceptance criteria, validation commands를 포함한다.
- snapshot field 변경 시 review는 stale로 표시된다.
- stale review는 unsafe action을 disable한다.
- stale 처리는 auto-review/auto-compile/auto-save/prompt mutation을 실행하지 않는다.
- quick capture/harness/project context guardrail은 유지된다.

### Task 10

- schema, grade utility, local reviewer, DB, IPC, UI tests가 통과한다.
- LLM 구현 시 missing key/malformed JSON/schema failure tests가 있다.
- guardrail scans가 forbidden production surface를 잡는다.
- 자동 LLM review/save/score update/prompt overwrite가 없음을 테스트한다.

### Task 11

- `npm run typecheck`, `npm run lint`, `npm test`가 통과한다.
- UI 구현 시 targeted Playwright/smoke tests가 통과한다.
- forbidden scans와 renderer isolation scans가 통과한다.
- manual QA checklist가 통과한다.

### Task 12

- atomic commit 순서는 contracts → local reviewer → DB persistence → IPC → LLM optional → UI → saved version integration → draft stale integration → guardrails 순서다.
- DB migration, renderer UI, LLM service를 한 commit에 섞지 않는다.

## 9. 보안 / 개인정보 / Guardrails

- API Key는 main process secret storage 내부에만 있다.
- renderer에 raw key를 반환하지 않는다.
- prompt body/API key를 log하지 않는다.
- prompt text는 secret을 포함할 수 있으므로 민감 데이터로 취급한다.
- Local review는 local-only다.
- LLM review는 명시적 클릭 후에만 prompt text를 OpenAI로 보낼 수 있고 UI가 이를 고지한다.
- Prompt Quality Reviewer는 prompt execution이 아니다.
- 외부 agent, command, git, repo scan, `repo_path` file read를 하지 않는다.
- `prompt_quality_reviews`는 review metadata 저장소이며 execution result 저장소가 아니다.
- Phase 11/12/13 guardrail을 깨지 않는다.

## 11. Draft 대비 변경점과 이유

### 추가

- MVP boundary를 명확히 추가했다.
  - 이유: draft는 LLM/history/export/improvement flow까지 넓어 scope creep 위험이 컸다.
- stale snapshot 규칙을 구체화했다.
  - 이유: 오래된 review를 현재 draft의 품질로 오해할 수 있다.
- compiler `qualityScore`, DB `quality_score`, review `overallScore`의 의미 충돌 해결 규칙을 추가했다.
  - 이유: hyperplan에서 기존 score semantics 충돌이 핵심 리스크로 지적됐다.
- export default exclusion 정책을 추가했다.
  - 이유: review metadata가 agent prompt export에 자동 포함되면 지시가 흐려질 수 있다.
- UI copy/provenance/score theater 방지 규칙을 추가했다.
  - 이유: 0-100 점수와 dimension bars가 거짓 정밀도를 줄 수 있다.
- forbidden surface scans를 추가했다.
  - 이유: Phase 14가 많은 계층을 건드려 Phase 11-13 guardrail regression 위험이 크다.

### 수정

- LLM review를 필수 중심 기능에서 explicit-click optional/stretch로 낮췄다.
  - 이유: secret handling, schema failure, privacy copy, external API behavior가 MVP를 막아서는 안 된다.
- `ambiguityRisk` 표시 방식을 수정했다.
  - 이유: risk score는 높을수록 나쁜데 일반 점수 bar와 나란히 두면 오해된다.
- improved prompt flow를 preview/new-version 중심으로 수정했다.
  - 이유: 저장된 versioned artifact의 자동 overwrite를 막아야 한다.
- prompt card/search score display를 optional qualitative badge로 축소했다.
  - 이유: score sorting/filtering은 query/UI scope를 크게 키운다.

### 삭제 / 보류

- review history 확장 UI, score sorting/filtering, export inclusion, bulk review, custom reviewer rules, marketplace/team features는 보류했다.
  - 이유: core reviewer가 검증되기 전에는 feature island가 된다.
- prompt execution, external agent launch, git execution, repo scan, run-result storage, global shortcuts, appEvents, shortcuts bridge, quick_capture settings는 명시적으로 scope에서 제거했다.
  - 이유: Phase 14의 명시적 금지사항과 Phase 11-13 guardrail을 위반한다.
