# Phase 10 최종 수정 계획

## 목표

Prompter를 실제 사용 가능한 macOS Electron 앱 수준으로 안정화한다. Phase 10은 새 제품 기능 개발이 아니라 기존 기능의 안정성, 테스트, 보안 경계, UI 마감, 키보드/메뉴, 패키징, 문서화를 완성하는 단계다.

계획 근거는 baseline-auditor, risk-auditor, logic-critic, creative-critic, deep-synthesizer가 Round 1 독립 검토, Round 2 상호 공격, Round 3 방어·정제·양보를 완료한 결과를 반영한다.

## 범위/비범위

범위:

- 기존 프롬프트 라이브러리, 버전, 검색, 태그, 설정, OpenAI 키 저장, 컴파일러, export 기능 안정화
- 순수 함수, Zod 스키마, export, diff, slug, compiler, repository, IPC 검증 테스트 보강
- 주요 UI의 loading, empty, error, saving, copied, exported, missing API key 상태 보강
- DESIGN.md와 기존 컴포넌트 래퍼를 따르는 접근성·사용성 polish
- Electron 메뉴와 macOS 중심 키보드 단축키 추가
- Electron 보안 경계 점검과 secret 노출 방지
- better-sqlite3, Drizzle migrations, userData DB, dev/prod DB 분리, macOS local package 검증
- README 또는 개발 문서와 docs/qa-checklist.md 작성

비범위:

- 프롬프트 실행 기능
- Codex, Claude Code, Cursor 등 외부 에이전트 실행
- Codex CLI, Claude Code CLI, Cursor 자동 실행
- prompt_runs, agent_runs, execution_results, validation_results, run_logs
- 실행 결과 저장, export history 저장
- cloud sync, 계정, 결제, 원격 서버, 팀 협업
- vector search, embedding 생성
- plugin system
- 자동 signing, notarization 강제 적용

## 현재 상태 감사

- package manager는 package-lock.json 기준 npm이다.
- 현재 scripts는 dev, build, build:electron, db:generate, native:electron, native:node, lint, typecheck, test, test:smoke가 있다.
- package 또는 make script는 아직 없다.
- Electron Forge 설정 파일은 확인되지 않는다.
- better-sqlite3는 native:node와 native:electron으로 재빌드 경로가 분리되어 있다.
- build는 typecheck, native:electron, build:electron, vite build 순서다.
- main process는 userData의 prompter.sqlite를 열고 app.getAppPath()/drizzle를 migration folder로 사용한다.
- drizzle migration 파일은 drizzle/와 drizzle/meta/에 존재한다.
- BrowserWindow 기본값은 contextIsolation true, nodeIntegration false, sandbox true로 설정되어 있다.
- preload는 contextBridge로 window.prompter만 노출하며 raw ipcRenderer를 직접 노출하지 않는다.
- renderer/src에서 Node, Electron, better-sqlite3, Drizzle 직접 import는 발견되지 않았다.
- IPC contract는 Zod payload/response schema 중심으로 구성되어 있다.
- 기존 테스트는 persistence, migration, contract, compiler, export formatter, diff, UI, smoke까지 이미 상당수 존재한다.
- Playwright smoke와 UI 테스트는 이미 있으므로 Phase 10에서 과도한 e2e 프레임워크 확장은 피한다.
- README.md는 현재 확인되지 않았으므로 Phase 10 문서 산출물로 추가 대상이다.
- docs/plan/plan_0010.md는 현재 비어 있으므로 이 문서가 최종 계획 본문 역할을 한다.

   - 완료 조건: userData DB를 테스트에서 건드리지 않고 temp DB만 사용한다.

   - 산출: 주요 화면의 누락 상태와 접근성 개선

   - 완료 조건: 인증서가 없으면 unsigned local build로 명확히 문서화한다.
8. 문서와 QA 체크리스트
   - 의존성: 1부터 7까지
   - 산출: README 또는 개발 문서, docs/qa-checklist.md
   - 완료 조건: MVP 기능, 제외 기능, 명령어, DB/secret 개요, packaging/signing 상태, 수동 QA가 포함된다.
9. 최종 검증과 정리
   - 의존성: 1부터 8까지
   - 산출: 실행 가능한 검증 결과 기록과 남은 blocker 목록
   - 완료 조건: 불가능한 검증은 정확한 blocker와 재시도 조건을 남긴다.

## 세부 작업 계획

### 1. 저장소 감사 확정

작업:

- package.json과 lockfile 기준으로 npm 사용을 확정한다.
- 현재 scripts와 누락된 package/make script를 기록한다.
- Electron packaging 도구가 없는 상태를 확인하고 도입 범위를 결정한다.
- better-sqlite3 native rebuild 흐름과 Electron target 일관성을 확인한다.
- drizzle migration 파일이 production package에 포함될 위치를 확인한다.
- renderer forbidden import를 재검사한다.
- OpenAI API key가 renderer, logs, export, settings table로 흐를 가능성을 점검한다.
- UI state, shortcut, app menu, external link 처리의 현재 gap을 정리한다.

인수 기준:

- 구현 전에 위험도와 작업 우선순위가 정리된다.
- 금지 기능과 허용된 작은 버그 수정 범위가 문서화된다.
- 패키징 도구 선택이 Phase 10 범위 안에서 최소 변경으로 결정된다.

### 2. 테스트 기반 정리

작업:

- compiler required sections, scenario, targetAgent, out-of-scope guardrail 테스트를 보강한다.
- export formatter가 secret과 run-storage term을 포함하지 않는지 확인한다.
- slug/filename 유틸이 한글, 공백, 특수문자, 긴 제목, 빈 제목 fallback을 처리하는지 테스트한다.
- diff utility의 추가, 삭제, 동일, 빈 문자열, 여러 줄 비교를 테스트한다.
- Zod schemas의 빈 문자열, enum, limit, export format, API key trim 검증을 보강한다.

인수 기준:

- 수정 전 실패하는 테스트를 먼저 추가한다.
- 큰 e2e 확장보다 빠르고 유지 가능한 unit/contract 테스트를 우선한다.
- 실제 OpenAI API key가 필요한 테스트를 추가하지 않는다.

### 3. persistence와 IPC 검증 보강

작업:

- 기존 temp SQLite 패턴을 재사용해 repository/service 테스트를 추가한다.
- createProject, listProjects, createPromptAsset, createPromptVersion, createNextVersion, setCurrentPromptVersion, createTag, attach/detachTagToPrompt, searchPrompts, rebuildSearchIndex, settings get/set 중 누락된 핵심 경로를 우선한다.
- IPC handler validation은 유지 가능한 범위에서 잘못된 입력 거부, 정상 입력 전달, 구조화된 에러 반환을 확인한다.
- 현재 구조상 IPC 테스트가 과해지면 repository/service 테스트를 우선하고 IPC는 핵심 schema boundary만 검증한다.

인수 기준:

- 테스트 DB는 temp directory에만 생성되고 정리된다.
- userData DB를 테스트에서 사용하지 않는다.
- prompt_runs 또는 실행 결과 관련 테이블이 생기지 않는다.

### 4. UI 상태와 접근성 polish

작업:

- 프로젝트 목록, 프롬프트 라이브러리, 프롬프트 상세, 컴파일러, clarification, 버전 목록, diff, 검색, 태그, 설정, export 흐름을 점검한다.
- loading, empty, error, saving, deleting, copied, exported, missing API key, invalid key format, LLM parse failure, no search result, no selection 상태를 빠진 곳에만 추가한다.
- 내부 stack trace, API key, DB path, secret file path가 UI에 표시되지 않도록 한다.
- Button, Input, Select, Textarea, Card, Tabs, EmptyState 등 기존 래퍼를 우선 사용한다.
- focus ring, label 연결, aria-label, keyboard navigation, disabled 상태를 점검한다.
- 세 패널 desktop shell을 유지하고 임의 collapse를 추가하지 않는다.

인수 기준:

- DESIGN.md 색상과 spacing token을 벗어나지 않는다.
- 새 대형 UI 패턴이나 새 디자인 시스템을 만들지 않는다.
- 사용자가 다음 행동을 알 수 있는 상태 메시지가 보강된다.

### 5. 키보드 단축키와 macOS 앱 메뉴

작업:

- Prompter, File, Edit, View, Window, Help 메뉴를 Electron menu로 구성한다.
- CmdOrCtrl+N, CmdOrCtrl+Shift+N, CmdOrCtrl+F, CmdOrCtrl+S, CmdOrCtrl+Shift+C, CmdOrCtrl+,, Esc 동작을 범위 제한적으로 연결한다.
- 메뉴 accelerator는 Electron menu에서 처리하고, renderer action이 필요한 경우 main에서 BrowserWindow.webContents로 좁은 이벤트만 보낸다.
- preload에는 허용된 menu action subscription API만 추가한다.
- renderer에는 typed action handler만 두고 raw ipcRenderer, shell, Node API를 노출하지 않는다.
- 입력 중 일반 타이핑을 방해하지 않도록 단축키 충돌을 제한한다.
- 개발 전용 reload/devtools 메뉴와 production 메뉴를 구분한다.

인수 기준:

- 메뉴 클릭과 단축키가 동일한 renderer action으로 연결된다.
- 동작 불가능한 상태에서는 데이터가 손상되지 않고 안전하게 무시되거나 사용자 상태가 유지된다.
- shortcut 추가가 보안 경계를 약화하지 않는다.

### 6. 보안과 logging hardening

작업:

- BrowserWindow 설정의 nodeIntegration false, contextIsolation true, sandbox true 유지 여부를 테스트로 고정한다.
- preload API 표면이 최소인지 확인하고 getOpenAIKey 같은 raw secret API를 만들지 않는다.
- renderer forbidden imports를 테스트 또는 정적 검사로 고정한다.
- export content와 logs에 API key, safeStorage payload, DB path, secret path가 포함되지 않도록 점검한다.
- settings table에 OpenAI API key 평문이 저장되지 않음을 확인한다.
- 외부 링크가 있다면 safe openExternal wrapper를 main process에 제한적으로 둔다.
- CSP는 Vite/Electron 개발 환경과 충돌하지 않는 범위에서 production 우선 적용을 검토한다.
- uncaught error는 secret 없는 사용자 친화 메시지와 개발용 detail을 분리한다.

인수 기준:

- secret은 main process 저장소 밖으로 raw value가 나오지 않는다.
- renderer는 Node, Electron internals, SQLite, Drizzle, filesystem에 직접 접근하지 않는다.
- CSP 적용이 개발 환경을 깨뜨리면 blocker와 대안 정책을 문서화한다.

### 7. 패키징 정리

작업:

- 현재 없는 packaging 도구를 최소 범위로 도입하거나 기존 Electron tooling에 맞는 package/make script를 추가한다.
- macOS 앱 이름 Prompter와 bundle identifier를 설정한다.
- better-sqlite3 native dependency가 packaged app에서 로드되는지 확인한다.
- drizzle migration files가 package에 포함되는지 확인한다.
- production 실행 시 app.getPath("userData")의 prompter.sqlite가 사용되고 dev DB와 혼동되지 않도록 한다.
- unsigned macOS local build를 기본값으로 문서화하고, signing/notarization은 인증서가 있을 때만 별도 단계로 남긴다.

인수 기준:

- package/make 명령이 정의된다.
- migration omission과 native rebuild failure를 조기에 잡을 수 있다.
- signing 인증서가 없다는 이유로 전체 local package가 깨지지 않는다.

### 8. 문서와 QA 체크리스트

작업:

- README 또는 개발 문서에 앱 설명, 기술 스택, 설치, 개발 실행, typecheck, lint, test, build, package 명령을 적는다.
- DB 위치, migration 개요, OpenAI key 저장 방식, 보안 원칙을 문서화한다.
- MVP 포함 기능과 명시적 제외 기능을 분리한다.
- packaging/signing/notarization 현재 상태를 명확히 적는다.
- docs/qa-checklist.md에 앱 시작, 프로젝트 생성, 프롬프트 생성, LLM analyze/compile, 저장, 버전 추가, current version 변경, 검색, 태그, export, 파일 저장, 설정, API key 삭제, 앱 재시작 후 데이터 유지, secret 비노출, prompt_runs 부재를 포함한다.

인수 기준:

- 새 개발자가 명령어와 보안 경계를 이해할 수 있다.
- QA 체크리스트는 사람이 순서대로 따라 할 수 있다.
- 금지된 기능을 구현하지 않았다는 확인 항목이 포함된다.

### 9. 최종 검증과 정리

작업:

- typecheck, lint, unit/contract tests, build, package/make, smoke/manual QA 순서로 검증한다.
- 실패 시 실패 명령, 원인, blocker, 재시도 조건을 기록한다.
- 변경 파일, 추가/삭제 패키지, 테스트 범위, 패키징 방식, signing/notarization 상태, 보안 점검 결과, 미구현 기능을 최종 요약한다.

인수 기준:

- 모든 검증 게이트가 통과하거나 정확한 blocker가 남는다.
- Phase 10 범위를 벗어난 기능이 추가되지 않는다.
- 최종 산출물이 재현 가능한 상태로 정리된다.

## 검증 게이트

1. 정적 검증
   - npm run typecheck
   - npm run lint
2. 낮은 위험 테스트

   - compiler, export, slug, diff, Zod schema 테스트 우선 확인
3. persistence와 IPC 검증
   - temp SQLite 기반 persistence 테스트

4. build 검증
   - npm run build
   - Electron main/preload bundle과 renderer build 확인
5. 패키징 검증
   - 추가된 package 또는 make script 실행

   - drizzle migrations 포함 확인
   - userData DB 생성 확인
6. smoke와 수동 QA

- 인증서 부재로 signing/notarization을 건너뛴 경우 local unsigned package와 배포 signing을 분리해 문서화한다.

## 리스크와 대응

- native rebuild 실패
  - 대응: native:node와 native:electron 경로를 분리 유지하고 package 환경에서 better-sqlite3 로드를 검증한다.
- migration 파일 package 누락
  - 대응: packaging 설정에 drizzle와 drizzle/meta 포함을 명시하고 packaged app에서 migration 실행을 확인한다.
- CSP가 Vite/Electron 개발 환경과 충돌
  - 대응: production 우선 적용 또는 blocker 문서화로 제한하고 개발 환경을 깨뜨리는 강제 적용을 피한다.
- 단축키 범위 과확장
  - 대응: Electron menu accelerator와 좁은 renderer action만 허용하고 입력 중 일반 키 동작을 막지 않는다.
- 실제 API key 테스트 유입
  - 대응: test client와 mocked status를 사용하고 real key나 secret 파일을 테스트 fixture에 저장하지 않는다.
- secret 노출
  - 대응: renderer, logs, export, settings table에서 API key와 secret path를 금지하는 테스트와 수동 점검을 둔다.
- scope creep
  - 대응: prompt execution, run history, cloud sync, accounts, vector search, plugin, collaboration은 Phase 10에서 명시적으로 금지한다.
- broad e2e 비용 증가
  - 대응: 이미 존재하는 Playwright smoke/UI 테스트를 유지하고, 새 검증은 unit/contract/temp DB 중심으로 추가한다.

## 수정/추가/삭제 요약

수정:

- 초안의 19개 광범위 체크리스트를 실행 가능한 9개 workstream으로 재구성했다.
- Phase 10 범위를 안정화, 테스트, 보안, polish, 패키징, 문서화로 더 엄격히 제한했다.
- 전체 smoke 우선 접근을 낮은 위험 테스트 우선 접근으로 조정했다.
- “가능하면 구현” 수준의 repository/IPC 테스트를 temp SQLite와 유지 가능성 기준으로 구체화했다.
- 키보드 단축키와 앱 메뉴를 Electron menu accelerator와 제한된 preload event bridge 중심으로 구체화했다.
- 패키징을 better-sqlite3, Drizzle migrations, userData DB, dev/prod DB 분리, unsigned local macOS build 중심으로 구체화했다.

추가:

- 구현 전 저장소 감사 게이트를 첫 단계로 추가했다.
- 작업 간 의존성과 ordered workstream을 추가했다.
- TDD 중심의 실패 테스트 우선 원칙을 추가했다.
- 원자적 커밋 전략을 추가했다.
- security hardening gate를 명확히 추가했다.
- README 또는 개발 문서와 docs/qa-checklist.md의 필수 내용을 추가했다.
- 검증 불가 시 exact blocker를 기록하는 규칙을 추가했다.
- Hyperplan 검토 출처를 간결한 provenance로 포함했다.

삭제 또는 축소:

- 새 대형 테스트 프레임워크 도입 암시를 삭제했다.
- broad mandatory e2e expansion을 축소했다.
- 자동 signing/notarization 강제 적용을 삭제하고 unsigned local build를 허용했다.
- prompt execution, external agent launching, run history, cloud sync, accounts, vector/embedding search, plugin system, team collaboration 관련 암시를 비범위로 고정했다.
- UI polish를 새 디자인 확장이 아니라 DESIGN.md와 기존 컴포넌트 래퍼 안의 누락 상태 보강으로 축소했다.

## 완료 기준

- Phase 10 구현 전에 저장소 감사 결과가 정리되어 있다.
- compiler, export, slug, diff, Zod schema 테스트가 보강되어 있다.
- 가능한 범위의 temp SQLite persistence 테스트가 있다.
- 유지 가능한 범위의 IPC validation 테스트가 있다.
- 주요 UI의 loading, empty, error, saving, copied, exported, missing API key 상태가 보강되어 있다.
- 접근성 label, focus, keyboard navigation이 기본 수준을 만족한다.
- macOS app menu와 핵심 keyboard shortcut이 보안 경계를 유지하며 동작한다.
- nodeIntegration false, contextIsolation true, sandbox true가 유지된다.
- renderer가 Node, Electron internals, SQLite, Drizzle, filesystem에 직접 접근하지 않는다.
- API key와 secret이 renderer, logs, export content, settings table에 노출되지 않는다.
- better-sqlite3와 Drizzle migrations가 packaged app에서 누락되지 않는다.
- dev DB와 production userData DB가 혼동되지 않는다.
- package/make 경로가 정의되어 있고, 인증서 부재 시 unsigned local build 상태가 문서화되어 있다.
- README 또는 개발 문서가 작성되어 있다.
- docs/qa-checklist.md가 작성되어 있다.
- typecheck, lint, test, build, package/make, smoke/manual QA 검증 경로가 준비되어 있다.
- 검증 불가 항목은 정확한 blocker와 후속 조치가 기록되어 있다.
- 프롬프트 실행, 외부 에이전트 실행, run history, cloud sync, accounts, vector/embedding search, plugin system, team collaboration 기능이 추가되지 않는다.
