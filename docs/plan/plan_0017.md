## Context

Target write path: `docs/plan/plan_0017.md`.

Phase 17 adds local library maintenance tools for Prompter: read-only scans, duplicate
detection, safe action previews, and explicitly confirmed maintenance repairs. The goal is to
help users understand and fix local database hygiene issues without changing prompt content,
running agents, scanning repositories, or adding background automation.

Current codebase facts to preserve:

- SQLite/Drizzle access belongs only in the Electron main process.
- Renderer code must use the typed `window.prompter` bridge and must not import Node,
  Electron internals, SQLite, Drizzle, filesystem APIs, path APIs, process, or environment
  variables directly.
- IPC contracts are centralized in `electron/ipc-contract.ts`, mirrored through
  `electron/bridge-types.ts`, `electron/bridge.ts`, and `electron/ipc-handlers.ts`.
- The current schema is in `electron/db/schema.ts`; repositories live under
  `electron/db/repositories/`; service composition lives in `electron/db/services.ts`.
- `prompt_asset_lineage` does not exist and must not be added.
- Prompt lineage is stored on `prompt_assets` via `parent_prompt_id`,
  `parent_prompt_version_id`, and `derivation_type`.
- `prompt_templates` exists and source references use `ON DELETE SET NULL`.
- `prompt_quality_reviews` exists and cascades with `prompt_versions`.
- `prompt_tags` uses foreign keys with cascade behavior.
- `project_context_profiles.project_id` cascades with projects.
- `SearchRepository.rebuildSearchIndex()` indexes only prompt assets joined to their current
  version. FTS health must treat search as a current-version projection, not an index of every
  `prompt_versions` row.
- `MENU_ACTION_CHANNEL` and `menu.onAction` exist and must be reused for any menu entry.

## Fixed Decisions

1. Phase 17 MVP is no-migration. Do not add `maintenance_reports`, archive columns, soft-delete
   columns, or new entity tables unless a later explicit phase asks for them.
2. Maintenance is a three-step model: `scanLibrary` -> `prepareAction` -> `executeAction`.
3. `scanLibrary` is read-only. It must not write to SQLite, mutate search indexes, call LLMs,
   read files, scan `repo_path`, run git, or persist reports.
4. `prepareAction` creates a main-owned maintenance action session. The renderer may request an
   action by id, but it must not send authoritative row mutations or dangerous payloads.
5. Action sessions are in main-process memory only, expiring, single-use, and invalidated after
   execute, cancel, failure, or a newer scan/action where appropriate.
6. Action sessions store the exact action type, selected entity ids, expected counts, relevant
   row snapshots or revision fields, warnings, and consequence summary needed for revalidation.
7. Destructive or relationship-changing actions require a backup recommendation and main-owned
   confirmation before execution. Renderer confirmation state is UX only and is not trusted as
   an authorization boundary.
8. Every DB-changing action revalidates entity existence, expected counts, snapshots, and action
   preconditions inside the same SQLite transaction as the mutation.
9. No async work is allowed inside SQLite transactions.
10. FTS rebuild is explicit/manual. It is not automatically run by scans and is not blocked by a
    stale scan session, but it still requires confirmation when invoked from Maintenance.
11. FTS health compares current searchable prompt assets to `prompt_search_fts` rows. It must not
    compare all prompt versions to FTS rows.
12. FTS rebuild must be atomic on the same SQLite connection as the maintenance operation that
    reports success.
13. Duplicate prompt detection is local heuristic only. MVP supports exact/normalized grouping,
    not fuzzy scoring, embeddings, vector DBs, or LLM judgment.
14. Duplicate prompt candidates are finding-only in MVP. There is no automatic prompt merge,
    automatic archive, or automatic delete.
15. Duplicate tag merge is allowed only after explicit canonical tag selection, preview, backup
    recommendation, and confirmation.
16. Unused tag delete is allowed only for selected tags that are still unused at transaction time.
17. Empty prompt asset delete is allowed only when the asset has zero versions, no child prompt
    references, no template source references, no quality reviews through versions, and no other
    relevant references. Otherwise it is finding-only with an explanatory warning.
18. `current_version_id` repair is allowed only when an asset has versions and the current pointer
    is null, missing, or points to a version that does not belong to that asset. Repair selects the
    highest `version_number` for the same asset.
19. Intentional `ON DELETE SET NULL` and `ON DELETE CASCADE` outcomes are not corruption. The UI
    should describe them as normal historical/source context loss when surfaced.
20. Broad orphan cleanup that is structurally prevented by current foreign keys is not an MVP
    action. It may appear only as guardrail tests or diagnostic findings if corruption can be
    deliberately created in test fixtures.
21. Template and harness issues are finding-first in MVP. Do not auto-rewrite template bodies,
    variables, JSON policy fields, or source references unless a narrowly proven impossible/corrupt
    state is observable under current schema.
22. Quality review maintenance is finding-only in MVP, except existing safe review deletion paths
    may be tested if current schema and repositories already support them without new tables.
23. The UI lives in Settings as a focused Maintenance workbench. A `Tools > Library Maintenance`
    menu item may be added only if it reuses the existing menu action channel to focus/open that
    Settings section.
24. Maintenance actions never mutate the prompt compiler draft, selected prompt content, current
    draft state, analysis answers, compiled output, or quality review drafts.

## Maintenance Model

The maintenance flow has three explicit phases.

1. Scan:
   - Renderer calls `window.prompter.maintenance.scanLibrary(input)`.
   - Main validates input through Zod.
   - Main reads current DB state and returns summary, findings, and action previews.
   - Scan returns only user-displayable summaries and entity ids needed for later selection.
   - Scan does not write, repair, rebuild, persist, or call external systems.

2. Prepare action:
   - Renderer calls `window.prompter.maintenance.prepareAction(input)` with a requested action
     type and selected ids from current scan results.
   - Main validates the request, recomputes the affected data, and creates a maintenance action
     session containing the exact consequence ledger.
   - Main returns a confirmation preview: action title, severity, destructive flag,
     affected counts, affected display names, warnings, backup recommendation, expiration, and
     confirmation requirements.
   - Prepare does not mutate data.

3. Execute action:
   - Renderer asks main to execute the prepared action session.
   - For destructive or relationship-changing actions, main shows a native/main-owned confirmation
     flow or equivalent main-owned confirmation gate before mutation.
   - Main revalidates all preconditions inside the write transaction.
   - If validation fails, the transaction rolls back and the result is a stale-plan failure.
   - On success, main invalidates the session and returns a result summary.
   - UI marks previous scan results stale and prompts/rescans after action.

## Scan Findings

### Duplicate Prompt Candidates

MVP detection:

- exact title match after trim.
- normalized title match using lowercase and whitespace/separator normalization.
- exact `original_input` match for current versions.
- exact `compiled_prompt` match for current versions.
- same normalized title plus same scenario and target agent.

Rules:

- Ignore blank normalized keys.
- Group only current-version prompt assets for prompt-body comparisons.
- Include project, scenario, target agent, updated time, version count, tag names, and quality score
  where available.
- Finding-only in MVP. The user may open/compare prompts, but Phase 17 does not auto-merge,
  archive, delete, or change prompts.

### Duplicate Tags

MVP detection:

- trim-identical names.
- case-insensitive names.
- separator-normalized names where spaces, hyphens, and underscores are treated consistently.

Action:

- User selects a canonical tag.
- Prepared action previews all prompt tag rewrites and duplicate tag deletes.
- Execute inserts missing canonical links with conflict protection, removes duplicate links, deletes
  duplicate tags, and refreshes tag/search-visible state.

### Unused Tags

Detection:

- Tags with zero `prompt_tags` links.

Action:

- User selects unused tags.
- Execute revalidates each selected tag is still unused before delete.
- Deleting tags never deletes prompts or prompt content.

### Empty Prompt Assets

Detection:

- Prompt assets with zero prompt versions.

Action:

- Delete only if transaction-time checks prove no versions, no children, no template source refs,
  and no other meaningful references.
- If any reference exists, show finding-only with an impact explanation.
- No archive in MVP.

### Current Version Issues

Detection:

- `current_version_id` is null while versions exist.
- `current_version_id` points to a missing version.
- `current_version_id` points to a version owned by a different prompt asset.

Action:

- Repair selected assets by setting current version to the highest `version_number` version owned by
  that asset.
- Rebuild/upsert FTS consistently in the same transaction.
- Assets with no versions are empty-asset findings, not current-version repair candidates.

### Search Index Health

Detection:

- Expected searchable set: prompt assets whose `current_version_id` joins to a version owned by the
  same asset.
- Compare expected prompt asset ids to `prompt_search_fts.prompt_asset_id`.
- Detect missing rows, extra rows, and mismatched indexed current-version text where practical.
- Detect missing or unusable FTS table as a high-severity finding.

Action:

- Explicit manual rebuild.
- Rebuild deletes and repopulates FTS from current-version projection atomically.
- If the FTS table itself is missing/corrupt beyond normal rebuild, return a clear failure rather
  than pretending success.

### Template And Harness Findings

Prompt Template findings:

- empty `template_body`.
- duplicate names with same scenario/target agent.
- source refs that are null should be displayed as historical source missing, not corruption.
- source refs that violate current schema should be treated as test-only corruption unless an
  observable production path exists.

Harness Template findings:

- empty `template_body`.
- invalid JSON in `required_fields` or `clarification_policy`.
- duplicate name/scenario/target agent groups.

Actions:

- Finding-only in MVP unless a narrow, schema-valid cleanup is proven during implementation.
- Do not auto-edit template bodies or JSON.

### Quality Review Findings

Detection:

- current version without quality score where latest review exists.
- latest review score differs from `prompt_versions.quality_score`.
- stale reviews may be listed as informational.

Actions:

- Finding-only in MVP.
- Do not automatically apply review scores or mutate prompt versions.
- Do not create prompt run or execution result concepts.

## Action Execution Rules

All write actions use a transaction and an action-specific revalidation plan.

Shared rules:

- Recompute targeted rows inside the transaction.
- Abort if any selected id no longer exists, unless the action explicitly treats disappearance as a
  stale-plan failure.
- Abort if counts or relevant row snapshots changed from the prepared plan.
- Abort if the action is no longer valid, such as a tag becoming used before unused-tag delete.
- Return structured errors without exposing stack traces, database paths, or internal SQL.
- Invalidate the action session after success, cancel, failure, or expiration.
- Mark scan results stale after any successful mutation.

Action-specific rules:

- `rebuild_search_index`: no entity ids required; manual confirmation; atomic rebuild from current
  version projection.
- `repair_current_versions`: selected assets only; choose highest same-asset version number; update
  FTS in the same transaction.
- `merge_duplicate_tags`: canonical id plus duplicate ids; canonical cannot be in duplicate ids;
  duplicate tags must still be duplicate candidates under the normalization rule; avoid duplicate
  prompt tag links.
- `delete_unused_tags`: selected tags must still have zero links at transaction time.
- `delete_empty_prompt_assets`: selected assets must still have zero versions and no child/source
  references at transaction time; delete prompt tag links and the asset in one transaction only if
  all checks pass.

## Draft-To-Final Change Notes

### Added

- Main-owned `prepareAction` / `executeAction` session model.
  - Reason: renderer-confirmed destructive payloads are not a safe authorization boundary.
- Transaction-time stale-plan revalidation rules.
  - Reason: scan results can become stale before user confirmation.
- FTS current-version projection rule.
  - Reason: existing search indexes current prompt assets, not every prompt version.
- Explicit no-migration MVP decision.
  - Reason: current schema can support the safe MVP without `maintenance_reports` or archive fields.
- Focused Settings Maintenance workbench and optional Tools menu focus action.
  - Reason: the draft's broad UI could bury dangerous actions in an unclear panel.

### Deleted

- `maintenance_reports` persistence from MVP.
  - Reason: report persistence is optional and expands schema/scope without being required for safe
    maintenance.
- Archive/soft-delete columns from MVP.
  - Reason: archive semantics would require migrations, list/search filtering policy, and wider UI.
- Fuzzy/Jaccard/Levenshtein/embedding/vector/LLM duplicate detection from MVP.
